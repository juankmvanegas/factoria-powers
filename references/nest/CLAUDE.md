## Architecture: Clean Architecture 3 Layers (ADR-001)

```
api → infrastructure → application
```

| Layer | Depends on | Contents |
|-------|-----------|----------|
| **api** | application | Controllers, Swagger decorators, Exception Filters, API contract (`contrato-de-api.yml`) |
| **infrastructure** | application | HTTP/gRPC clients to backend microservices, HTTP Error Interceptor |
| **application** | NOTHING | Services (use cases), Abstractions (`abstract class`), DTOs, Enumerations |

Plus `libs/` for shared utilities: config (AppSettings, Key Vault), tracer (CustomLogger, TracerInterceptor), errors (custom exceptions), observability (OpenTelemetry SDK).

### Non-Negotiable Rules

- Exactly 3 layers. No layers are created or renamed.
- application has ZERO external dependencies on other layers.
- application ONLY defines abstractions (`abstract class` — NOT TypeScript `interface`, because NestJS DI needs runtime tokens) and DTOs.
- infrastructure implements the abstractions defined in application.
- api depends ONLY on application (never on infrastructure directly).
- Controllers for REST (NO direct route handlers without controllers).
- DI based on `abstract class` tokens always. Provider pattern: `{provide: AbstractClass, useExisting: ConcreteImpl}`.
- Services return `Observable<T>` (RxJS) — NOT `Promise<T>`. The entire stack is reactive.
- Each layer registers its own providers in its module.
- `libs/` is shared across all layers — it contains cross-cutting utilities only.

### BFF Golden Rule

The BFF must **ONLY**:
- Aggregate or transform data from multiple backend sources
- Orchestrate calls to backend microservices
- Validate input data (DTOs with class-validator)
- Verify Authorization header presence and extract claims (without validating token authenticity — APIM validates JWT against Azure AD B2C)
- Handle errors and exceptions

The BFF must **NOT**:
- Implement complex business rules
- Process business data or domain calculations
- Persist data in databases (no ORM, no direct DB access)
- Make domain-level decisions

### Authentication Model (ADR-004)

APIM (Azure API Management) sits in front of the BFF and validates the JWT token against Azure AD B2C. The BFF:

1. **Does NOT validate** token authenticity (signature, expiration, issuer) — APIM already did this
2. **Extracts claims** from the `Authorization` header for routing/context purposes
3. **Verifies presence** of the Authorization header (returns 401 if missing)
4. **Never stores** tokens or secrets in memory beyond the request lifecycle

### Initialization Type (ADR-004)

The BFF has a single initialization type: **REST API** (NestJS application via `main.ts`).

```typescript
// main.ts — Single entry point
const app = await NestFactory.create(InitializationModule);
app.useGlobalPipes(new ValidationPipe({ transform: true }));
await app.listen(port);
```

### Execution Order for New Features

1. application — Abstractions (interfaces), DTOs, Enumerations, Services (use cases)
2. infrastructure — HTTP/gRPC clients implementing abstractions, `infrastructure.module.ts`
3. api — Controllers, Swagger decorators, Exception Filters, `api.module.ts`
4. libs/ — Shared utilities if needed (config, tracer, errors)
5. Tests — Mocks, Unit tests
6. Documentation — CHANGELOG, architecture updates

> **Nota**: El blueprint usa archivos sin sufijo de tipo (e.g., `notes.ts` en vez de `notes.controller.ts`). El documento nestjs.txt prescribe sufijos. **Seguir la convención del nestjs.txt** (con sufijos) para nuevos proyectos, pero **respetar la convención existente** al modificar proyectos ya creados con el blueprint.

### TypeScript Conventions

- Path aliases: `@api/*`, `@infrastructure/*`, `@application/*`, `@libs/*`
- PascalCase for classes, camelCase for methods/properties, SCREAMING_SNAKE_CASE for constants
- DTOs de entrada: `class` con decoradores `class-validator` (`@IsString()`, `@IsNumber()`, etc.)
- DTOs de salida: `type` alias (NO class) — e.g., `export type NoteOutput = { id: string; ... }`
- No `any` — use proper typing or `unknown` when truly necessary
- No `console.log` — only `CustomLoggerService`
- No commented-out code

### Abstraction Pattern (CRITICAL)

NestJS requiere tokens de DI en runtime. TypeScript `interface` se borra en compilación, por lo tanto:

**SIEMPRE usar `abstract class` para abstracciones, NUNCA `interface` de TypeScript:**

```typescript
// application/abstractions/use-cases/notes.ts
// CORRECTO — abstract class sirve como token de DI
export abstract class NotesUseCases {
  abstract createNote(data: NoteCreationInput): Observable<NoteOutput>;
  abstract getAllNotes(): Observable<SimplifiedNoteOutput[]>;
}

// INCORRECTO — interface se borra en runtime, no funciona para DI
// export interface INotesUseCases { ... }
```

### Reactive Pattern (RxJS Observables)

Todo el stack es reactivo. Los servicios retornan `Observable<T>`, NO `Promise<T>`:

```typescript
// Servicio de aplicación — retorna Observable
@Injectable()
export class NotesService implements NotesUseCases {
  constructor(private notesAdapter: NotesAndListsAdapter) {}

  createNote(data: NoteCreationInput): Observable<NoteOutput> {
    return this.notesAdapter.createNote({ Title: data.title }).pipe(
      map((data: Note) => ({
        id: data.Id,
        title: data.Title,
        state: data.State,
      }))
    );
  }
}
```

### Provider Registration Pattern

```typescript
// Infraestructura — provider con useExisting para vincular abstract class → implementación
```

### Error Handling

8 tipos de excepciones personalizadas en `libs/errors/`:

| Excepción | HTTP Status | Contexto |
|-----------|------------|----------|
| `HttpBusinessError` | 409 | Error de negocio desde backend HTTP |
| `HttpInternalError` | 500 | Error interno desde backend HTTP |
| `HttpNotFoundEntityError` | 404 | Recurso no encontrado en backend HTTP |
| `GrpcBusinessError` | 409 (gRPC code 9) | Error de negocio desde backend gRPC |
| `GrpcInternalError` | 500 (gRPC code 13) | Error interno desde backend gRPC |
| `GrpcNotFoundEntityError` | 404 (gRPC code 5) | Recurso no encontrado en backend gRPC |
| `BadGatewayError` | 502 | Backend inaccesible (ECONNREFUSED) |
| `InitializationError` | - | Error al arrancar (Key Vault, config) |

**Flujo de errores**:
1. Backend (HTTP/gRPC) falla → `HttpErrorInterceptor` (infraestructura) traduce a excepción tipada
2. Excepción tipada llega al `CustomExceptionFilter` (api) → respuesta segura al cliente
3. `CustomLoggerService` registra el error internamente con stack trace completo
4. **PROHIBIDO**: exponer detalles de sistema, stack traces, o nombres de servicios al cliente

**Estructura de respuesta de error**:
```json
{ "statusCode": 409, "message": "Business Error In Backend (Grpc)", "conflictDetail": "..." }
```

## Third-Party Integrations (Backend Microservices)

Since the BFF orchestrates calls to backend microservices, integrations are its primary function.

### Where to store backend service documentation

```
.cloud/contracts/backend-services/
├── {service}-openapi.yaml     ← Backend microservice spec (if provided)
├── {service}-endpoints.md     ← Textual documentation (if no spec)
└── mapping.md                  ← Spec → generated infrastructure services mapping
```

### How to generate infrastructure services from backend documentation

Follow the Clean Architecture execution order:

**1. Application — Abstractions and DTOs**
```typescript
// application/abstractions/infrastructure/i-{service}.interface.ts
export interface I{Service}InfraService {
  {operation}(input: {Request}Dto): Promise<{Response}Dto>;
}

// application/dtos/{service}/{request}.dto.ts
// application/dtos/{service}/{response}.dto.ts
```

**2. Infrastructure — HTTP implementation**
```typescript
// infrastructure/services/{service}/{service}.service.ts
@Injectable()
export class {Service}InfraService implements I{Service}InfraService {
  constructor(
    private readonly httpService: HttpService,
    private readonly configService: ConfigService,
  ) {}
  // Base URL from ConfigService (NEVER hardcoded)
  // API keys/tokens from Azure Key Vault (NEVER in code)
}
```

**3. Module Registration**
```typescript
// infrastructure/infrastructure.module.ts
providers: [
  {
    provide: 'I{Service}InfraService',
    useClass: {Service}InfraService,
  },
],
```

### Based on what is received

| What is received | What is generated |
|-----------------|------------------|
| OpenAPI/Swagger | Interface + DTOs faithful to the schema + Complete HTTP implementation |
| Textually described endpoints | Interface + DTOs based on description + HTTP implementation |
| Nothing (placeholder) | Interface with methods `// TODO: completar cuando la documentacion este disponible` + Empty DTOs + Stub implementation that throws `NotImplementedException` |

### Rules for backend microservice integrations

- **NEVER** hardcode backend service URLs — always `ConfigService` → `configService.get('BACKEND_SERVICES.{service}.baseUrl')`
- **NEVER** hardcode API keys/tokens — always Azure Key Vault
- **ALWAYS** use `HttpService` from `@nestjs/axios` for HTTP calls
- **ALWAYS** implement retry/timeout configuration for backend service calls
- **ALWAYS** generate unit tests mocking the infrastructure service interface
- **ALWAYS** propagate the Authorization header to backend microservices when required

### Configuration Rules

- **NEVER** commit secrets, connection strings, or API keys to the repository
- **ALWAYS** use environment variables for environment-specific configuration

### Rules

- **AAA** pattern (Arrange-Act-Assert) mandatory
- Naming: `[feature].test.ts` (e.g., `notes.test.ts`)
- `describe` blocks for grouping: `describe('NotesService', () => { ... })`
- `it` blocks with descriptive names: `it('should return all notes when backend service responds successfully', () => { ... })`
- One behavior per test
- Jest mocking for dependencies (`jest.fn()`, `jest.spyOn()`)
- `@golevelup/ts-jest` for NestJS-specific mocking (`createMock<T>()`)
- Mock data in `test/datasets/mocks/`, shared across tests
- Unit tests target ONLY Application services (application layer)
- Architecture tests validate layer dependencies with `dependency-cruiser` (ADR-012)
- **90% minimum coverage** — CI/CD enforces this gate
- CI/CD order: Architecture → Unit
- Test configuration in `jest.config.ts` at project root

### Test Example

```typescript
// test/unit-testing/services/notes.test.ts
import { Test } from '@nestjs/testing';
import { createMock } from '@golevelup/ts-jest';
import { NotesService } from '../../../src/application/services/notes.service';
import { INotesInfraService } from '../../../src/application/abstractions/infrastructure/notes.interface';
import { notesMock } from '../../datasets/mocks/notes.mock';

describe('NotesService', () => {
  let service: NotesService;
  let infraService: INotesInfraService;

  beforeEach(async () => {
    const module = await Test.createTestingModule({
      providers: [
        NotesService,
        {
          provide: 'INotesInfraService',
          useValue: createMock<INotesInfraService>(),
        },
      ],
    }).compile();

    service = module.get<NotesService>(NotesService);
    infraService = module.get<INotesInfraService>('INotesInfraService');
  });

  it('should return all notes when infrastructure service responds', async () => {
    // Arrange
    jest.spyOn(infraService, 'findAll').mockResolvedValue(notesMock);

    // Act
    const result = await service.findAll();

    // Assert
    expect(result).toEqual(notesMock);
    expect(infraService.findAll).toHaveBeenCalledTimes(1);
  });
});
```

## Security (Absolute Priority Policy)

- **NO secrets in code** — everything in Azure Key Vault
- No system details in error messages
- Validation with class-validator at API boundaries (controllers)
- Sensitive data encrypted in transit (TLS 1.2+, AES-256, RSA 2048+)
- Hash: only SHA-2 or SHA-3
- No hardcoded or shared credentials
- Inactivity timeout: maximum 10 minutes
- Protection against injection (SQL, XSS, NoSQL)
- Production inaccessible from development
- Do not mix dev/QA with production
- Authorization header: verify presence, extract claims — NEVER validate token (APIM's responsibility)
- Compliance: Colombian law, ISO 27000, NIST SP800-50

## Auto-Shielding (Self-Healing System)

When an error occurs:
1. **FIX** — Correct the code
2. **TEST** — Verify it works
3. **DOCUMENT** — Record the learning
4. **NEVER HAPPENS AGAIN**

### Where to document

| Type | Location |
|------|----------|
| Feature-specific | "Learnings" section of the current PRP |
| Multi-feature | Relevant skill in `.claude/skills/*/SKILL.md` |
| Critical systemic | This file (`CLAUDE.md`) |

### Format

```markdown
### [YYYY-MM-DD]: [Short title]
- **Error**: What exactly failed
- **Fix**: How it was resolved
- **Apply to**: Where else it applies
```

## Automatic Chain

After ANY code change:
```
Code → [security-scan] (always) → verify-logic (if there is legacy) → /generate-tests (auto) → /documentacion (auto)
```

### In migration context:
```
migration-execute generates code → verify-logic → /generate-tests → /documentacion
```

Logic verification, tests, and documentation run AUTOMATICALLY. No manual invocation is needed.

### How Auto-Activation Works

When a skill completes its execution, it invokes the next one in the chain:

1. `/migration-execute` → invokes `verify-logic` with the migrated module
2. `verify-logic` (if approved) → invokes `/generate-tests` with the verified services
3. `/generate-tests` (upon completion) → invokes `/documentacion` automatically
4. `/documentacion` → updates CHANGELOG, architecture, README. End of chain.

Each skill WAITS for the previous one to complete. If any fails, the chain stops and is reported.

## Sub-Agents (.ai/agents/)

| Agent | Responsibility |
|-------|---------------|
| orchestrator-agent | Coordinates all sub-agents |
| discovery-agent | Analyzes legacy BFF code, extracts contracts and orchestration patterns |
| architecture-agent | Technical decisions, generates ADRs |
| migration-agent | Executes migration (one module at a time) |
| testing-agent | Generates and validates tests |
| docs-agent | Updates documentation |
| execution-agent | Implements features (non-migration work) |
| planning-agent | Plans complex features, generates PRPs |

### Chain Rules

- Sub-agents NEVER call each other — only the orchestrator invokes them
- Each sub-agent receives only the context it needs
- The orchestrator waits for completion before invoking the next one
- Approval gates: after discovery, after plan, between each module

## Migration Workflow (Mandatory Order)

```
Step 0: /migration-start
  ↓ Capture constraints, architecture changes
  ↓ Generate migration-constraints.md + new ADRs
  ↓ Team confirms

Step 1: /migration-discovery [legacy path]
  ↓ discovery-agent extracts contracts and orchestration patterns
  ↓ Creates files in .cloud/planning/legacy-discovery/
  ↓ Team reviews and validates

Step 2: /migration-plan
  ↓ architecture-agent confirms decisions + generates ADRs
  ↓ Creates .cloud/planning/migration-plan.md
  ↓ Team explicitly approves

Step 3: /migration-execute [module name]
  ↓ rollback-plan generates snapshot and reversion plan
  ↓ migration-agent migrates ONE module
  ↓ verify-logic compares against the original legacy
  ↓ Coverage >= 90%? → fix gaps until met
  ↓ validate-contracts verifies API compatibility
  ↓ /generate-tests runs automatically
  ↓ smoke-tests verifies the service actually works
  ↓ /documentacion updates docs
  ↓ audit-trail records everything
  ↓ One module at a time, team approves each one
```

**Rules**: Do not skip steps. No batch. No silent gaps. No tests without logic verification. Everything through the orchestrator.
