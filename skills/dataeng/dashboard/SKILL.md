# Skill: Dashboard

## Purpose

Track progress of large projects (like Omega with 50 services). A single command to see where everything stands. Without having to open 20 files, without having to remember what was done and what remains.

The dashboard is the project's control center. It shows the truth based on real files, not assumptions.

---

## Execution Flow

### Phase 1: Project State Scan

Read ALL relevant files in `.cloud/planning/`:

1. **Constraints and Configuration**
   - `migration-constraints.md` — General migration state, constraints
   - `migration-plan.md` — Planned modules and execution order

2. **Discovery**
   - `legacy-discovery/` — Modules discovered in the legacy
   - For each module: what was found (APIs, services, data, etc.)

3. **Logic Verification**
   - `logic-verification/` — verify-logic reports per module
   - Logic coverage percentage per module

4. **Smoke Tests**
   - `smoke-tests/` — Smoke test results per module
   - Pass/fail per module

5. **Contract Validation**
   - `contract-validation/` — Contract validation results
   - Breaking changes found/approved

6. **Rollback**
   - `rollback/` — Active rollback plans
   - Executed rollbacks

7. **Audit**
   - `../audit/` — Audit trail entries
   - Recent events

8. **Documentation**
   - Existing ADRs
   - CHANGELOG
   - Architecture documentation

### Phase 2: Metrics Calculation

For each identified module, determine its current phase:

| Phase | Criterion |
|-------|----------|
| Discovery | Exists in legacy-discovery/ |
| Planned | Appears in migration-plan.md |
| In Code | Code files exist for the module |
| Logic Verified | Report exists in logic-verification/ with >= 95% |
| Contracts Validated | Report exists in contract-validation/ without pending breaking changes |
| Tests | Unit tests exist and pass |
| Smoke Tests | Report exists in smoke-tests/ with PASS |
| Documented | ADR exists, documentation updated |
| COMPLETED | All previous phases ✅ |

Calculate:
- Total modules
- Completed modules
- In-progress modules (in some intermediate phase)
- Pending modules (only discovery or plan)
- Modules with problems (some phase failed)
- Overall progress percentage

### Phase 4: User Presentation

Show concise summary directly in the conversation:

1. **If migration is in progress**:
   - Show per-module progress table
   - Highlight active problems
   - Recommend next action

2. **If NO migration is in progress**:
   - Show general project health score
   - Show documentation status
   - Show test status
   - Recommend improvements

---

## Strict Rules

1. **ALWAYS** read from real files — never guess or cache the state
2. **Update the dashboard every time it is invoked** — the dashboard always reflects the current state
3. **Can be invoked at ANY time** during any workflow
4. **If no migration is in progress**, show general project health metrics
5. **Do not modify other files** — the dashboard only reads and generates its own report
6. **Be honest with the numbers** — if a module has problems, show it clearly
7. **Include timestamps** — each dashboard update must have date and time
8. **Maintain history** — if possible, show trends (improved/worsened since last time)
9. **The dashboard is the visual source of truth** — if something does not appear here, it does not exist for the project
10. **Prioritize actionable information** — the user should know WHAT TO DO NEXT when viewing the dashboard

---

## Operation Modes

### Migration Mode
When migration files are detected in `.cloud/planning/`:
- Show per-module progress with all phases
- Focus on the migration pipeline
- Highlight blockers and breaking changes

### General Project Mode
When there is NO active migration:
- Show general health score
- Show test status and coverage
- Show documentation status
- Show known technical debt

### Specific Module Mode
If the user specifies a module: `/dashboard [module]`
- Show deep detail of that module only
- Include all associated reports
- Include audit-trail event timeline for that module

## Integration with Other Skills

- **All skills feed the dashboard** — each skill generates files that the dashboard reads
- **health-check**: The health score is shown in the dashboard as a summary
- **audit-trail**: Latest events are shown in the dashboard
- **rollback-plan**: Active and executed rollbacks are shown
- **smoke-tests**: Smoke test results per module
- **validate-contracts**: Pending breaking changes are highlighted
- **verify-logic**: Logic coverage percentage per module
